
# Samokat To-Do Service

RESTful HTTP-сервер для управления списком задач (To-Do), разработанный в рамках тестового задания для **Самоката**. Сервис написан исключительно на **стандартной библиотеке Go** (без использования сторонних фреймворков и библиотек) с соблюдением принципов чистой архитектуры.

## 🚀 Особенности реализации

* **Zero Dependencies**: Использована только стандартная библиотека Go (включая продвинутый роутинг `net/http` с поддержкой методов и паттернов, доступный в Go 1.22+).
* **Clean Architecture**: Четкое разделение проекта на независимые слои:
  * `domain` - основные модели данных и бизнес-ошибки.
  * `repository` - слой работы с хранилищем (в данном случае In-Memory).
  * `service` - слой бизнес-логики и валидации.
  * `transport/http` - HTTP-хендлеры, маршрутизация и DTO.
* **Thread-Safe In-Memory Storage**: Кастомное хранилище на основе `map` со срезом (`slice`) для гарантированного сохранения порядка добавления задач. Потокобезопасность реализована через `sync.RWMutex`.
* **Продвинутый Middleware Chain**: Запросы проходят через цепочку кастомных middleware:
  * **RequestID**: Извлекает или генерирует уникальный идентификатор запроса (`X-Request-ID`), прокидывая его в контекст для сквозного трейсинга.
  * **Logger**: Логирует метод, путь, HTTP-статус, размер ответа в байтах и реальное время выполнения запроса.
  * **Recover**: Безопасно перехватывает паники, логирует стек вызовов и возвращает клиенту статус `500 Internal Server Error`, не позволяя серверу упасть.
* **Production-Ready HTTP Server**:
  * Настроены жесткие таймауты для защиты от медленных запросов (`ReadTimeout`, `WriteTimeout`, `IdleTimeout`).
  * Защита от перегрузки памяти: размер тела входящего запроса ограничен до 1 МБ с помощью `http.MaxBytesReader`.
  * **Graceful Shutdown**: При получении сигналов ОС (`SIGINT`, `SIGTERM`) сервер прекращает принимать новые запросы и плавно завершает обработку текущих соединений в рамках выделенного таймаута.
* **Контекстные таймауты**: Все операции внутри хендлеров контролируются через `context.WithTimeout`, предотвращая зависание горутин.
* **Добавлена пагинация**: На эндпоинте GET /todos можно указать limit и offset

## 🛠 Конфигурация

Сервис поддерживает конфигурирование через переменные окружения (значения по умолчанию зашиты в Dockerfile и конфигурационный слой):

| Переменная окружения | Описание | Значение по умолчанию |
| :--- | :--- | :--- |
| `HTTP_ADDR` | Адрес и порт для запуска HTTP-сервера | `:8080` |
| `HTTP_REQUEST_TIMEOUT` | Таймаут на обработку одного HTTP-запроса | `3s` |
| `HTTP_SHUTDOWN_TIMEOUT`| Максимальное время для плавной остановки сервера | `10s` |

## 📦 Запуск приложения

### Вариант 1. С использованием Docker (Рекомендуемый)

Убедитесь, что у вас установлен Docker. Соберите образ и запустите контейнер из корневой директории проекта:

```bash
# Сборка Docker-образа
docker build -t samokat_todo .

# Запуск контейнера с пробросом портов
docker run -p 8080:8080 samokat_todo
````

### Вариант 2. Локальный запуск через Go CLI

Убедитесь, что у вас установлен Go версии **1.22** или выше.

Bash

```
# Запуск сервера напрямую из корня проекта
go run ./cmd
```

## 📑 Спецификация API (Endpoints)

### Структура объекта задачи (Todo)

JSON

```
{
  "id": 1,
  "title": "Купить продукты",
  "description": "Молоко, хлеб, сыр",
  "completed": false,
  "created_at": "2026-05-22T17:30:00Z",
  "updated_at": "2026-05-22T17:30:00Z"
}
```

### Список эндпоинтов

#### 1. Создать новую задачу

- **Запрос**: `POST /todos`
    
- **Тело запроса**:
    
    JSON
    
    ```
    {
      "title": "Изучить Go",
      "description": "Посмотреть новые фичи паттернов в ServeMux",
      "completed": false
    }
    ```
    
- **Коды ответов**:
    
    - `201 Created` — Задача успешно создана, возвращается объект с присвоенным ID.
        
    - `400 Bad Request` — Пустой заголовок (`title` не передан или состоит из пробелов) либо невалидный JSON.
        

#### 2. Получить список всех задач

- **Запрос**: `GET /todos`
    
- **Параметры пагинации (Query)**: `?limit=20&offset=0` _(По умолчанию: limit=50, максимальный limit=100)_
    
- **Коды ответов**:
    
    - `200 OK` — Возвращает массив объектов (сохраняя хронологический порядок добавления).
        

#### 3. Получить задачу по идентификатору

- **Запрос**: `GET /todos/{id}`
    
- **Коды ответов**:
    
    - `200 OK` — Задача найдена, возвращается объект.
        
    - `400 Bad Request` — Невалидный ID (не является числом или `id <= 0`).
        
    - `404 Not Found` — Задача с указанным ID не существует в хранилище.
        

#### 4. Обновить задачу по идентификатору

- **Запрос**: `PUT /todos/{id}`
    
- **Тело запроса**: (структура аналогична созданию)
    
- **Коды ответов**:
    
    - `200 OK` — Задача успешно обновлена, поле `updated_at` актуализировано.
        
    - `400 Bad Request` — Ошибка валидации (пустой `title`) или неверный формат ID.
        
    - `404 Not Found` — Задача для обновления не найдена.
        

#### 5. Удалить задачу по идентификатору

- **Запрос**: `DELETE /todos/{id}`
    
- **Коды ответов**:
    
    - `204 No Content` — Задача успешно удалена из памяти, тело ответа отсутствует.
        
    - `400 Bad Request` — Некорректный формат ID.
        
    - `404 Not Found` — Задача с указанным ID отсутствует.
